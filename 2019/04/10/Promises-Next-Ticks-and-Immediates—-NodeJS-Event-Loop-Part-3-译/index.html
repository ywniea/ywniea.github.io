Native Promises

原生的JS promise的callback会被当做是一个microtask放到microtask queue中，event loop会在清空了next tick queue之后去处理microtask queue中的任务。

然后event loop开始轮询：

原生JS中next tick queue的优先级要高于microtask queue。因此两个都不为空的时候先处理next tick queue中的event。

  • event loop检查发现next tick queue中有3个event，于是就处理这三个，直到队列清空。
  • 然后event loop发现promise所在的microtask queue中有5项内容，于是就去执行这些resolved promise的callback。在执行到第三个的时候，process.nextTick向next tick queue中添加了一个event。一直处理microtask queue中的event直到队列清空。
  • 然后event loop又检测到了next tick queue不为空。因为它在执行promise的callback的时候又被添加了一项到next tick queue中。于是又去清空next tick queue。
  • 现在next tick queue和microtask queue都是空的了。event loop终于可以开始第一个阶段的轮询了：timer queue。timer queue中有一项已经到达预期时间的timer的callback，于是event loop就执行了这个callback。
  • 在保证中间队列都是空的情况下，event loop到了下个阶段：IO queue。因为IO queue是空的也没有正在等待的IO事件，event loop又跳到了immediates queue，发现里面有四项内容，于是就依次处理这四个回调函数，直到清空immediates queue。
  • 最后，所有队列里面都没有event，这个轮询就结束了。

BlueBird.resolve().then的callback与setImmediate的callback语义是一样的，因此bluebird的callback会被schedule到immediates queue中。并且会部署在在原生的immediate之前。

Q.resolve().then的callback与process.nextTick的语义是一样的，q的callback会被schedule到next tick queue中。并且部署在原生的next tick task之前。

BlueBird.resolve().then callback has the same semantics as the following setImmediate call. Therefore, bluebird’s callback is scheduled in the same immediates queue before the setImmediate callback. Since Q uses process.nextTick to schedule its resolve/reject callbacks, Q.resolve().then is scheduled in the nextTick queue before the succeeding process.nextTick callback.

在新版本的NodeJS中使用setImmediate代替process.nextTick是有优势的。因为自从NodeJS v0.12以后就没有process.maxTickDepth了，过多的向next tick queue中添加event会导致IO饥饿，反而使用setImmediate就安全的多。因为immediates queue是在IO queue之后的。所以setImmediate永远不会导致IO饥饿。
